# 代码生成复查规则

> 本 skill 在需求实现后强制触发（R-FORCE-002）。  
> 复查分两类，**严格区分**——决定性项自动修，启发式项**仅报告不擅自改**（R-FORCE-003）。

## 复查时机

当 AI 根据需求文档（MD 文件、用户描述或 Issue）生成了一批代码后，在向用户报告完成之前必须执行。

**适用范围**：
- 需求涉及 3 个以上文件的改动时，必须执行
- 单文件小改动（如修复 typo、调整样式）可以跳过
- 用户明确要求跳过复查时可以跳过
- **纯 markdown / 文档改动**：只跑确实适用的检查项；其他不适用的项**汇总为一行**（如 `[N/A] Java 决定性项：纯文档改动不涉及`），不要逐条列 N/A

## 两轮复查结构

| 轮次 | 目标 | 处理策略 |
|---|---|---|
| 第一轮 | 功能完整性 | 对照需求逐条检查，遗漏立即补 |
| 第二轮 | 代码质量合规性 | 决定性项自动修，启发式项仅报告 |

## 第一轮：功能完整性

详见 `references/functional-completeness.md`。核心 6 项：

1. 需求逐条对照
2. 输入输出异常完整性
3. 桩代码 / TODO 检测
4. 前后端接口对齐
5. 配置项 / 常量遗漏
6. 发现遗漏立即补

## 第二轮：代码质量合规性

### A. 决定性检查（Auto-Fix）

**有明确规则可执行，发现即改。** 详见 `references/deterministic-checks.md`。

清单：

- [ ] **字段 `@Autowired` → setter 注入**（R-FORCE-001 / R-BE-002）
- [ ] 未使用的 import → 删除
- [ ] 硬编码 magic number → 提取常量（仅当对应常量类已存在）
- [ ] 命名违反 R-NAME-001~003 → 重命名（仅本次新增的符号）
- [ ] 缺少类注释 / 方法注释 → 按模板补
- [ ] Repository / Mapper / Provider / Entity / Info 命名后缀错误 → 改

### B. 启发式检查（Report Only）

**需要人判断，仅报告，不擅自改。** 详见 `references/heuristic-checks.md`。

清单：

- [ ] 方法长度 > 50 行 → 标注，建议用户考虑拆分
- [ ] 文件长度 > 500 行 → 标注
- [ ] 命名是否表意 → 列出可疑项给用户决策
- [ ] 是否存在 N+1 查询 → 标注嫌疑位置
- [ ] 是否需要新增异常处理 → 列出可能漏处理的路径
- [ ] 注释是否描述了 WHY 而非 WHAT → 列出疑似冗余
- [ ] 是否需要拆分为更小的 Service → 列出疑似职责过重

> 启发式项的处理：**列清单 + 给用户决策**，禁止把这些当成"必须修复"硬改。这是 Karpathy "Surgical Changes" 原则的本地落地。

## 输出要求

复查结果以下列三段清单返回：

```markdown
## 复查报告

### 第一轮：功能完整性
- [✓] <检查项> ：通过
- [→] <检查项> ：已补（说明）
- [!] <检查项> ：存在问题（附说明）

### 第二轮 A：决定性检查（已自动修复）
- [→] <检查项> ：已修（说明）
- [✓] <检查项> ：通过

### 第二轮 B：启发式检查（仅报告，请用户决策）
- [?] <检查项> ：可疑/建议（位置 + 说明）
```

## 不做的事

- 不在启发式项上擅自修改代码
- 不删除"原本就存在"的死代码（除非用户确认）
- 不重构未被需求触及的方法
- 不修改未被本次改动触及的文件的格式 / 注释
